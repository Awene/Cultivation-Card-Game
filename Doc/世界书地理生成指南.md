# 世界书地理生成指南

本文档是基于"本格修仙·北境"实施过程沉淀的经验，为后续生成新地域(中原/南疆/东土/西域/灵界/仙界)提供结构、写作、技术、命名的统一规范。**新地域生成必须先读本文档**。

---

## 一、核心架构（务必遵循）

### 推荐: 总览自包含 + 各 entry 独立输出

```
[lorebook entry: 区域总览]      ← 蓝灯 区域名 关键字
  自身 EJS 内构造完整 brief 输出
  含: 地理列表 / 宗门列表 / 凡国列表 / (区域内时附加)核心地形/小宗/地图

[lorebook entry: 生态.txt × N]   ← 绿灯 具体生态名 关键字
[lorebook entry: 宗门.txt × M]   ← 绿灯 宗门名 + 人物名 等
[lorebook entry: 凡国.txt × K]   ← 绿灯 国名 + 都城 + 王称 等
  各自 EJS 内构造 detail outputText, 末尾 <%- outputText %> 输出
  当玩家提及关键字时激活, 输出独立 section 到 prompt
```

**核心原则**: 单个 entry 自给自足，**不依赖跨 entries 的变量传递**。总览始终输出全部 brief，子 entry 提到时附加 detail（brief 与 detail 共存于 prompt，重复一些但稳定）。

### 不要使用的架构（已验证不可靠）

❌ **跨 entries setvar/getvar 通信**：初始化 entry setvar 默认值 → 子 entries setvar 同名 key 覆盖 → 总览 getvar 读取拼接。理由见"四、失败教训"。

## 二、关键技术点

### 1. ST 宏与 EJS 的处理顺序（极重要）

环境：SillyTavern + Tavern Helper + ST-Prompt-Template 扩展。

- **ST 宏**: `{{setvar::A::B}}`、`{{getvar::A}}`、`{{ if }}`、`{{random}}` 等
- **EJS 标签**: `<%_ ... _%>`、`<% %>`、`<%- %>`、`<%= %>`

**实测顺序**：
- 蓝灯 always-on 类 entries：EJS 通常先于 ST 宏（待验证，至少在本项目可行）
- **绿灯关键字触发 entries：ST 宏先于 EJS**

**结论**：
- ❌ 永远不要把 EJS 标签放进 ST 宏的 value 中
  - `{{setvar::A::<%- expr %>}}` → ST 把 `<%- expr %>` 当字面量存进 var，EJS 永远不解析
- ❌ 不要把 ST 宏放进 EJS template literal 中（除非已确认稳定）
  - `outputText = \`{{setvar::A::${value}}}\`` → 也会因 ST 宏处理时机早而失败
- ✅ 推荐：**EJS 内做所有事情，最终用 `<%- outputText %>` 输出纯文本**

### 2. ST-Prompt-Template 变量 scope 体系

| scope | 实际存储 | 持久 | ST 宏 `{{getvar::}}` 可读 |
|---|---|---|---|
| `global` | `extension_settings.variables.global` | ✅ 跨 chat | 通过 `{{getglobalvar::}}` |
| **`local`** | `chat_metadata.variables` | ✅ 当前 chat | ✅ **`{{getvar::}}` 默认读这里** |
| `message` | `chat[id].variables[swipeId]` | ✅ 当前楼层 | ❌ |
| `cache` | `STATE.cacheVars` (内存) | ❌ 临时 | ❌ |
| `initial` | `STATE.initialVariables` | ✅ 初始值 | ❌ |

**注意函数默认 scope**：
- `setvar(name, value)` 默认 scope='message' ← 与 `{{getvar::}}` 不通
- `getvar(name)` 默认 scope='cache' ← 与 `{{setvar::}}` 不通
- **使用 `setLocalVar(name, value)` 或显式 `setvar(name, value, { scope: 'local' })`**

### 3. EJS 块语法

```ejs
<%_ // 修剪前后空白的 scriptlet (推荐) _%>
<% // 不修剪空白的 scriptlet %>
<%- expr %>  // 不转义输出 (中文/markdown 推荐)
<%= expr %>  // HTML 转义输出
```

EJS 内可调用：
- `getvar(name)` / `setvar(name, value)` / `setLocalVar(name, value)` (ST-Prompt-Template)
- `getMessageVar(path)` / `setMessageVar(name, value)` (Tavern Helper)
- `getwi(entry_name)` / `activewi(entry_name)` (ST-Prompt-Template)
- 全局 `_` (lodash)、`$` (jquery) 等
- 普通 JS：模板字符串、Math、Array 方法等

### 4. 推荐输出模式（标准模板）

```ejs
<%_
// 1. 取数据
const world = getMessageVar('stat_data.地点.世界') || '凡界';
const region = getMessageVar('stat_data.地点.地域') || '';
const 具体地点 = getMessageVar('stat_data.地点.具体地点') || '';
const year = Number(getMessageVar('stat_data.时间.年')) || 7203;

// 2. 计算
const inRegion = (region === '北境');
const nextOpen = (anchor, period, y) => y < anchor ? anchor : anchor + (Math.floor((y - anchor) / period) + 1) * period;
const pick3 = (arr) => [...arr].sort(() => Math.random() - 0.5).slice(0, 3).join('、');

// 3. 数据字典 (本 entry 自包含, 不跨 entries)
const 动物 = ['...', '...'];
const 植物 = ['...'];
const 矿产 = ['...'];

// 4. 拼装 outputText
const outputText = `### 标题
- 位置: ...
- 地貌: ...
- 资源:
  - 动物: ${pick3(动物)}
  - 植物: ${pick3(植物)}
  - 矿产: ${pick3(矿产)}`;
_%>
<%- outputText %>
```

## 三、写作约定

### 1. 区域分级

| 层级 | 含义 | 示例 |
|---|---|---|
| 世界 | 凡/灵/仙界 | 凡界 |
| 地域 | 大区(7个左右) | 北境 / 中原 / 南疆 |
| 生态 | 大区内子区域 | 雪松峡湾 / 银沙落星漠 |
| 具体地点 | 城市/秘境/地标 | 鲸髓城 / 沉船浮港 |

stat_data.地点 路径: `stat_data.地点.世界` / `.地域` / `.具体地点`

### 2. 修仙境界 (凡界)

**凡界最高: 化神后期**（含化神前/中/后期）。无渡劫、无合体、无大乘。

层级:
- 练气 → 筑基 → 金丹(初/中/后期) → 元婴(初/中/后期) → 化神(初/中/后期)

### 3. 物品品级

五等 (从低到高): **凡 < 黄 < 玄 < 地 < 天**

❌ **不在提示词中标注品级**（用户决定）。
✅ 在 Doc 内部记录或注释中可标注，便于平衡。

### 4. 秘境

每生态 **2-3 个秘境**。每秘境分两态：

#### Brief (默认显示)
```
- 秘境名: 一句话简介 (周期性: "下次开启:XXXX年X月" / 常驻: "(常驻)")
  入口: 入口位置 + 进入条件
  内部地点: 区域A / 区域B / 区域C / ...
```

#### Detail (玩家在秘境内 OR 提到秘境名时)
```
- 秘境名 :
  顺序: 描述 (线性 1→2→3 / 分支 1→{2a/2b}→3)
  1. 区域名 [境界] — 一行环境描述
    险: 风险描述
    解: 应对方案
    获: 战利品列表 (可标注随机/孕养/炼器至宝等功能描述, 不标品级)
  ...
```

**禁忌**:
- ❌ 提示词中不出现"层"字眼 (用"区域"/"地段"/"地点")
- ❌ 周期性秘境的周期 < 5 年 (避免每年开)
- ❌ "渡劫"/"合体"/"大乘"等凡界不存在的境界

**秘境周期计算**：
```js
const nextOpen = (anchor, period, year) =>
    year < anchor ? anchor : anchor + (Math.floor((year - anchor) / period) + 1) * period;
```
anchor (起点年) 与 period (周期) **隐藏在代码中**，提示词仅显示 `下次开启:XXXX年`。

### 5. 凡国

每地域 **5-8 个凡国**。每凡国 brief 含: 名 + (地形分类) + 简介一行 + 组织摘要。
detail 含: 简介 / 风格原型 / 位置 / 政体 / 都城 / 立国年份 / 组织(王/议会/职官) / 军事 / 经济 / 宗教 / 节庆 / 禁忌 / 对外关系 / (可选)特殊设定。

凡国 brief **始终显示**（不嵌套在 inRegion 条件里），因为凡人国家是公开信息。

### 6. 宗门

每地域 **3-5 个大宗 + 1-3 个小宗**。

- **大宗** (含"修仙城" 这种修真据点): 始终显示 brief，含主要修士姓名/境界/性别
- **小宗** (隐世/独立散修宗): **仅 inRegion 时显示**

宗门 brief 含: 宗名 + 道统 + 简介 + 主要人物(2-5 位, 含姓名/性别/境界/职位)
detail 含: 完整组织 / 地标 / 外销 / 对外关系 / 特殊技艺

### 7. 地图

mermaid graph TD 拓扑图，**仅 inRegion 时输出**。包含:
- 大区下属生态(矩形/圆形/菱形分类)
- 主要凡国与宗门驻地
- 关键通行路径(实线/虚线/粗线分类)

### 8. 命名（极重要 - 已踩坑）

每个枚举数组(动物/植物/矿产/海族) **数组内不允许同前缀重复**。例:
- ❌ ['极光鳐', '极光狐'] 同数组 2 个"极光"
- ❌ ['月华苔', '月华针'] 同数组 2 个"月华"
- ✅ 跨数组(动/植/矿)同前缀 1 次可接受
- ✅ 跨生态共有种(海洋类如 巨鲸/寒蛟)允许

**命名风格**：参考山海经、古典文献。优先用 玄/朱/烛/鲲/鹏/魍魉/螭/蜮/胤/麒麟/夔/英招 等山海经异兽词。避免现代造词(如"风暴"+"X"系列泛滥)。

每数组规模 **10-12 项**(海洋"海族"标签代替"动物"，禁含职位词如"斥候"/"使者")。

新地域生成后，将其完整字典追加到 [Doc/北境字典.md](北境字典.md) 同级文件(如 `中原字典.md`)，包含跨地域查重表。

## 四、失败教训汇总（避免重蹈覆辙）

### ❌ 失败 1: ST 宏外包 EJS 标签
```ejs
<%_ const v = compute(); _%>
{{setvar::北境地理-X::<%- v %>}}
```
**为何失败**: ST 宏处理顺序优先于 EJS（至少在绿灯关键字触发的 entries 中）。ST 把 `<%- v %>` 当字面量存进 var，EJS 永远不会解析。
**症状**: prompt 中看到字面量 `<%- v %>` 字符串。

### ❌ 失败 2: outputText 模式仍内嵌 ST 宏
```ejs
<%_
const outputText = `{{setvar::A::${value}}}`;
_%>
<%- outputText %>
```
**为何失败**: 同上，ST 宏处理时机仍可能早于 EJS 输出。在某些蓝灯 entries 上能 work，绿灯不行。

### ❌ 失败 3: setvar 默认 scope 不是 local
```ejs
<%_ setvar('A', value); _%>  // 默认写到 message scope
```
然后总览 `{{getvar::A}}` 读不到 (它读 local scope = chat_metadata.variables)。
**修正**: 用 `setLocalVar('A', value)` 或 `setvar('A', value, { scope: 'local' })`。

### ❌ 失败 4: 跨 entries 的 setLocalVar→getvar 通信
即使所有 setLocalVar 都正确写到 chat_metadata.variables，跨 lorebook entries 通信仍不可靠：
- entries 的 EJS 处理顺序 不一定按 lorebook 顺序
- 总览 entry 的 outputText 含 `{{getvar::A}}` 字面，ST 解析时机可能早于子 entries 的 setLocalVar 调用
- **重新导入角色卡（新 chat）**时 chat_metadata 完全为空，更彻底失效

**症状**: 第一次 work 是因为之前残留 var 缓存，重导后所有 brief 段为空。

### ✅ 成功 5: 自包含模式
- 总览 entry 的 EJS **内部计算所有 brief**，不依赖任何外部 var
- 各子 entry 直接 `<%- outputText %>` 输出 detail 到 prompt（独立 section）
- brief 与 detail 共存于 prompt（重复但不冲突）
- 重导角色卡也不影响

## 八、调试与验证

### 1. 看 prompt 实际输出
SillyTavern → 设置 → 用户设置 → 启用 "Show generated prompt" 或类似选项。生成后查看 prompt 文件是否符合预期。

### 2. 检查 EJS 是否运行
在 EJS 块末尾加测试输出：
```ejs
<%_
// ... 你的代码
_%>
<%- '\n[DEBUG: 此 entry EJS 已运行]\n' %>
```
若 prompt 中看到此字符串，EJS 跑了。若没看到，EJS 未运行（lorebook entry 可能未激活，检查关键字与 yaml）。

### 3. 检查 var 写入
在 ST console (F12) 执行：
```js
window.SillyTavern.getContext().chatMetadata.variables
```
查看实际写入的 var。如果有期望的 key，则 setLocalVar 工作了。

### 4. 重导角色卡验证
**最重要的验证步骤**：在新 chat / 重导角色卡后第一次生成时验证。这能发现"残留 var 缓存"导致的伪成功。

### 5. 标准测试矩阵

| 玩家位置 | 期望 prompt 内容 |
|---|---|
| 凡界·中原(其他大区) | 总览 brief 全部，无核心地形/小宗/地图 |
| 凡界·中原 + 提到具体生态名 | + 该生态 detail (独立 section) |
| 凡界·北境 | 总览 brief + 核心地形 + 小宗 brief + 地图 |
| 凡界·北境 + 具体地点含秘境名 | + 该生态 detail (秘境段切到 detail 多区域探索) |
| 灵界/仙界 | 北境完全不输出 |

## 十、本项目历史经验小结

| 阶段 | 尝试 | 结果 |
|---|---|---|
| v1 | EJS `{{setvar::A::<%- 段 %>}}` | ❌ ST 宏字面化 EJS 标签 |
| v2 | EJS `outputText = \`{{setvar::A::${v}}}\`` | ❌ 仅蓝灯部分 work, 绿灯失败 |
| v3 | EJS `setvar('A', value)` | ❌ 默认 message scope, {{getvar}} 读不到 |
| v4 | EJS `setLocalVar('A', value)` | ⚠️ 旧 chat 残留缓存时 work, 重导失败 |
| **v5** | **总览 EJS 自包含全部 brief, 子 entry 直接 `<%- outputText %>` 输出** | ✅ **稳定** |

记住：**单 entry 自包含 + 直接输出 > 跨 entries var 通信**。
