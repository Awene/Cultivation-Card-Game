# 事件写作规范

> 给宗门/世界写**多段分支剧情事件**（宗门主线、奇遇、秘境事件等）的统一规范与脚手架。
> 参照实现：命定之诗《双子星的咏叹调》（解读见对话记录）。骨架模板：[[模板]事件骨架.txt](../世界书/事件/[模板]事件骨架.txt)。
>
> 这是「宗门细化」的**第二步**——[宗门细化工作流程.md](宗门细化工作流程.md) 第一步只写生活与修炼，主线剧情用本规范单独实现、单独挂载。

---

## 一、本质：一个变量驱动的状态机

一个事件 = 一个世界书条目（常驻/蓝灯），内部按 `stat_data.事件.阶段` 分成许多 `if` 块。

**运转三要素**：

| 角色 | 是谁 | 干什么 |
|---|---|---|
| 世界书条目 | 只读的"当前页" | 每轮重新求值，`阶段` 是多少就只渲染那一段剧本 |
| `事件.阶段` 变量 | "书签" | 记录当前演到哪 |
| LLM | "翻页的手" | 演完当前段后，按剧本里的指令用 `<UpdateVariable>` 把 `阶段` 拨到下一格 |

**推进不是代码做的，是 LLM 做的**——世界书只负责喂"当前该演的那一段"，"演完翻页"是写在剧本里、命令模型执行的。下一轮条目重新求值，新段亮起。这是整套设计的核心，务必理解。

## 二、变量（已接入项目）

事件状态存在 `stat_data.事件` 下，已加进 [[initvar]变量初始化默认禁用.yaml](../世界书/[initvar]变量初始化默认禁用.yaml) 与 [[mvu_update]变量更新规则.yaml](../世界书/[mvu_update]变量更新规则.yaml)：

```yaml
事件:
  开启: false        # 当前是否有事件进行中
  标题: ""           # 进行中事件名(空=无)
  阶段: ""           # 状态机指针,当前演到哪
  已完成事件: []      # 已通关事件名列表,用于一次性门禁
```

- **`阶段` 是唯一的程序计数器**，取值就是你起的中文阶段名（`第一幕`→`抉择`→`结局·甲`…）。
- **`已完成事件`** 用来防止一次性事件重复触发。
- 这些变量走的是项目既有的 `<UpdateVariable>` + JSON Patch (RFC 6902) 框架，和数值更新完全同源，不是另一套引擎。

## 三、剧本里的三种记号

写 `<event>` 正文时，三种记号性质完全不同，别混：

| 记号 | 含义 | 例 |
|---|---|---|
| 普通文字 | **照抄输出**的定稿旁白 | `尽头并列着两张王座` |
| `${……}` | **给模型的生成指令**（不是字面文字，模型据此现编） | `莉莉姆："${反驳对方的怜悯}"` |
| `<!-- -->` / `[标签]` | **导演注释 / 场景节拍**，指导结构、不直接演 | `<!--华丽特效-->`、`[对话冲击]` |

`${}` 是"半成品填空"：骨架钉死（谁说话、什么情绪、推进到哪），血肉每轮由模型即兴——既保证走向不跑偏，又让每次体验不同。**能定死的氛围用普通文字，需要随玩家变化的用 `${}`。**

## 四、四个必备结构件

### 1. 触发器（一次性开闸）

```js
if (标题 !== 本事件 && !已完成.includes(本事件) && /* 你的条件 */ ) {
  // 开场叙述 + UpdateVariable 置 开启=true / 标题=本事件 / 阶段=首阶段
}
```

三重门禁：**没在进行 && 没通关过 && 满足触发条件**。触发条件常用取法（模板已备好）：
- 身份：`身份文本.includes('五毒')`（本宗弟子）
- 境界：`L >= 2`（筑基以上）
- 地点：`具体地点.includes('神木殿')`
- 也可挂道具/时间/前置事件是否完成。

### 2. 状态机主体（阶段块）

```js
if (标题 === 本事件) {          // 总闸: 只有本事件进行中才渲染
  if (阶段 === '第一幕') { <event>...</event> }
  if (阶段 === '抉择')   { <event>...分支...</event> }
  if (阶段 === '结局·甲') { <event>...发奖+归档...</event> }
}
```

每块结尾必须有推进指令：

```
本轮必须在<UpdateVariable>中更新(变量不可见,必须照做):
  将'事件.阶段'的值设置为 `下一阶段名`
```

### 3. 分支（选择即自由文本解读）

选择**不是按钮**，是把所有选项列出来、让模型根据玩家那句自由输入判断命中哪条：

```
**若{{user}}选择甲**
本轮仅描写: ${甲的短过场}
本轮必须在<UpdateVariable>中更新: 将'事件.阶段'的值设置为 `结局·甲`

**若{{user}}选择乙**
本轮仅描写: ${乙的短过场}
本轮必须在<UpdateVariable>中更新: 将'事件.阶段'的值设置为 `结局·乙`
```

### 4. 结局 + 兜底阀门

- **结局块**：收束叙述 + 发一件带完整属性的奖励（把叙事结局翻译成机制收益）+ 归档：
  ```
  将'事件.已完成事件'数组中 add 一项 `本事件名`
  将'事件.开启'设为 false; '事件.标题'设为 ``; '事件.阶段'设为 ``
  ```
- **兜底阀门**（每个事件必写）：玩家出格致剧情演不下去时，强制归档收尾。没有它，玩家一旦走出预设格子，状态机就卡死。

## 五、分支设计的三条经验（来自参照实现）

1. **能合流就合流**。命定之诗里 `失衡的天平-莉莉姆` 和 `-塞拉` 走不同剧情，但都汇入同一个 `幕间`——分支张开后尽快收束，否则剧本数量指数爆炸。
2. **只在框架不同处才复制**。到最终三选一时两条线内容几乎一致，就直接复制两个近似块，不必强行合并。
3. **用一个"元叙事角色"托举抉择**。命定之诗的"幻灵"是个游戏内角色看不见、只对玩家说话的旁白者，负责在抉择点打断、点破、催促——比干巴巴的 `[请选择]` 沉浸得多。你的项目可以用一个类似的引导者（或不用，直接旁白）。

## 六、落地步骤（写一个新事件）

1. **复制模板**：把 [[模板]事件骨架.txt](../世界书/事件/[模板]事件骨架.txt) 复制到 `世界书/事件/<事件名>.txt`。
2. **改事件名**：把文件里的 `本事件 = '示例事件'` 改成你的事件名；同步改所有 `UpdateVariable` 指令里的事件名。
3. **填触发条件**：删掉触发器里的 `&& false` 占位行，写真实条件。
4. **画状态图再动笔**：先把阶段流转（含分支、合流、结局）在纸上画成图，再逐块填 `${}` 与旁白。
5. **发奖与归档**：每个结局块补奖励物品 + 归档指令；补兜底阀门。
6. **注册条目**：在 [本格修仙.yaml](../本格修仙.yaml) 里加一段（**蓝灯常驻**，和 `[战斗规则]` 同款，每轮求值、EJS 自行决定是否输出）：
   ```yaml
     - 名称: "[事件]<事件名>"
       启用: true
       激活策略:
         类型: 蓝灯
       插入位置:
         类型: 指定深度
         角色: 系统
         深度: 2
         顺序: 6          # 比战斗规则(11)略靠前, 让当前剧本更受重视; 可调
       激活概率: 100
       递归:
         不可被其他条目激活: false
         不可激活其他条目: true
       文件: 世界书\事件\<事件名>
   ```
7. **验证**：抽出 EJS 主体用 node 跑，逐个喂 `阶段` 值，确认：①无事件时空输出 ②每阶段只渲染自身 ③他事件进行中静默 ④已通关不复燃。参考脚本：`scratchpad/test_event.mjs`。
8. **登记**：把该事件填进 [宗门设定清单.csv](宗门设定清单.csv) 的「是否有主线剧情 / 主线剧情简介」两列。

## 七、注意事项

- **蓝灯常驻、EJS 自控输出**：事件条目每轮都求值，靠内部 `if(标题===本事件)` 决定出不出内容；无事件时输出空串，不占 token。这一选择已被命定之诗卡验证——见「十」的三代写法对比，我们的脚手架 = 第二代（蓝灯常驻）。**不要用绿灯关键字**去触发状态机主体（关键字会漏、会误触）。
- **阶段名唯一且稳定**：`阶段` 字符串必须与 `if` 判断、`UpdateVariable` 指令三处完全一致，错一个字整台机器卡住。
- **每块必写推进指令**：忘写 `UpdateVariable` → 玩家永远卡在这一阶段。
- **每个事件必写兜底阀门**：玩家一定会出格。
- **变量不可见**：`事件.*` 是幕后指针，指令里要反复强调"变量不可见、必须照做、不要在正文叙述它"。
- **多轮阶段**：像"幕间"这种需要跨多轮慢慢演的阶段，在块里写 `<!--本阶段跨越多轮回复-->`，并把推进指令放在"进行数日日常描写**之后**才触发"，避免一轮就冲过去。
- **只用既有设定**：地标、人名、势力来自现有世界书；奖励物品用项目既有的品质/类型/效果格式。

## 八、两种变量更新模式（来自命定之诗卡）

命定之诗卡把"谁来写变量"做成了**可切换的两种模式**，靠启用两个 `output_format` 条目中的**哪一个**来决定。两者用的是**同一个 `<UpdateVariable>` + JSON Patch 契约**，区别只在"由哪个模型发、发的时候带不带正文"。

| 模式 | 对应条目（都在 order 10000 · 蓝灯 · depth0） | 谁更新变量 | 输出形态 |
|---|---|---|---|
| **随主API**（默认开） | `output_format (随AI输出开，主API)` | 讲故事的主模型 | 一条回复里：`正文 + <UpdateVariable>…</UpdateVariable>` |
| **外部额外API**（默认禁） | `[mvu_update]output_format (使用额外模型更新变量开)` | 另调一个副模型专职记账 | 只输出 `<UpdateVariable>`，**禁止输出任何正文**（"DO NOT output anything else"） |

- **切换方式**：两个条目二选一启用，另一个禁用。不能同时开。
- **随主API**：一个模型既讲故事又更新变量——简单、省一次调用，但变量更新会和创作抢注意力，长文时偶尔漏更或更错。
- **外部额外API**：主模型只管讲故事（它那份 `output_format` 里就没有变量指令），另用一个便宜/稳定的小模型，喂它剧情上下文、只让它吐 `<UpdateVariable>`。**变量准确性显著提升、可用弱模型省钱**，代价是每轮多一次 API 调用、要配外部模型管线。
- **对事件的意义**：事件靠 `<UpdateVariable>` 推进 `阶段`。**两种模式下事件脚本一字不用改**——你只管在 `<event>` 里写"本轮必须更新 阶段=…"，至于这条指令是被主模型还是副模型执行，由模式开关决定，对事件透明。

> 本项目现状 = **随主API**（[[mvu_update]变量输出格式（随主AI输出开）.yaml](../世界书/[mvu_update]变量输出格式（随主AI输出开）.yaml) 就是主API那份）。要上外部模式，照命定之诗加一个"只输出 UpdateVariable、不写正文"的 output_format 条目，与现有那份二选一启用即可。

## 九、变量设计进阶（来自命定之诗卡）

几个我们项目值得吸收的点：

1. **JSON Patch 操作集更丰富**。命定之诗支持 `replace / delta / insert / remove / move`，比我们现有的 `replace / add / remove` 多了三个，其中最有用的是：
   - **`delta`**：数值增量。`{"op":"delta","path":"/资源池/气血/现值","value":-30}` ——直接扣 30，不用先读旧值算新值。对 HP/灵气/好感度这类频繁增减的量是质变，模型不必做算术、也不会算错。
   - **`insert`**：往对象/数组插入，数组用 `-` 表示追加到末尾（`/事件/已完成事件/-`）。
   - **`move`**：搬运（如驯服后把妖兽从"关系列表"移到某人名下）。
   - 若要吸收，改 [[mvu_update]变量输出格式（随主AI输出开）.yaml](../世界书/[mvu_update]变量输出格式（随主AI输出开）.yaml) 的操作集声明即可（这是给模型看的契约，不是引擎限制）。

2. **`事件.信号`（string[]）—— 信号总线**。除了单一的 `阶段` 指针，命定之诗还有一条 `事件.信号` 数组：事件往里 `insert` 命名信号（如 `学院扩展_card_received`、`密室_completed`），其他条目用 `signals.includes(SIG_X)` 各自反应。
   - 用途：**非线性、多入口、跨条目联动**的事件——不同碎片监听不同信号，而不是挤在一条线性 `阶段` 上。
   - 省 token 技巧：信号的**更新规则本身是条件注入**的——`事件信号变量适配` 条目只有当 `事件.信号` 已是数组时才输出那段规则文本，没有信号时完全不进上下文。
   - 若要吸收，把 `信号: []` 加进 `事件` 变量结构即可（我们当前脚手架用 `阶段` 就够，信号是可选增强）。

3. **EJS 可以直接写变量**（`setMessageVar` / `setvar`）。命定之诗新一代事件在 EJS 里直接 `setMessageVar('stat_data.事件.信号', [])` 做初始化，而不是每次都指望模型去建。
   - 意义：**幂等的初始化/建容器这类活儿交给代码，不占模型的更新预算、也不会被模型忘掉**。玩法数据（阶段推进、发奖）仍走模型的 `<UpdateVariable>`（因为要基于剧情判断），但"确保容器存在"这种确定性操作可以代码直接做。
   - 注意区分：`getMessageVar`/`setMessageVar` 与 `getvar`/`setvar` 是两套 API 变体，命定之诗里混用；本项目统一用 `getMessageVar`（与现有总览一致）。

4. **两条硬约定**（照抄）：`_` 前缀变量只读、绝不更新；需要"空"时给**具体空值**（`""`/`[]`/`{}`），**禁用 `null`**。

## 十、事件实现的三代写法（选型参考）

同一张命定之诗卡里并存三代事件写法，越新越灵活：

| 代 | 触发/驱动 | 例子 | 特点 |
|---|---|---|---|
| **一代** | 绿灯 + 关键字 `当前事件为<标题>` + `阶段` 字符串 | 双子星的咏叹调、被遗忘者 | 本体是绿灯条目，需一个桥接把"当前事件为X"注入上下文才能每轮触发；`生成结果示例.txt` 就是这代 |
| **二代** | **蓝灯常驻 + EJS 门控 `if(标题===本事件)` + `阶段`** | 斯芬克斯、冰之歌、神选者、瓷化密室 | 无需桥接、无关键字漏触发；可加 `Math.random()<p` 概率激活。**我们的脚手架 = 这代** ✅ |
| **三代** | **蓝灯常驻 + `事件.信号` 总线 + 命名信号** | 学院扩展（学生证/入学分院） | "积木式"：多个碎片各监听信号、非线性；适合开放式/多入口玩法 |

**选型建议**：
- **线性主线剧情**（如五毒教《药引之争》：空寨→两派→火并→结局）→ 用**二代**（`阶段` 状态机，就是我们脚手架）。
- **非线性/多触发点/需跨条目联动**（如一个宗门里多条支线互相影响）→ 用**三代**（`事件.信号`，在脚手架基础上加 `信号` 数组）。
- **不要用一代**——绿灯+桥接是历史包袱，二代已完全取代它。

> 一句话：**我们的脚手架已对齐命定之诗的第二代主流写法**；等真需要非线性玩法时，再把 `事件.信号` 加上去升到三代即可，无需推倒重来。
