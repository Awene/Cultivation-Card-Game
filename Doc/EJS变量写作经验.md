# 经验教训：EJS 变量声明冲突（`Identifier 'xxx' has already been declared`）

> 日期：2026-07-12
> 涉及：本格修仙预设条目（`路由隐藏规则` / `[NSFW]` 门控片段）、命定之诗世界书

## 一、现象

玩家在游玩中途（"玩了一百多层突然碰到，前面都正常"）弹出报错：

```
EJS Error
Identifier '_hn' has already been declared ... while compiling ejs
... <%_ let _hn; try { _hn = JSON.parse(getvar('路由隐藏规则') || '[]'); } ...
   <%_ let _hn; try { _hn = JSON.parse(getvar('路由隐藏规则') ...
at line: 228, column: 9
```

注意报错片段里 `<%_ let _hn;` **连续出现了两次**——这是关键线索。

## 二、根因

**酒馆助手会把整轮激活的提示词拼接成「一个」EJS 模板统一编译**，所有 `<% %>` 代码块共享**同一个函数作用域**。

- `let` / `const` 在同一作用域内**不允许重复声明**，第二次声明直接抛 `already been declared`。
- 我们多条条目（如「📑行动选项」「🔥NSFW最适配/男性向/女性向」「🔥NSFW词汇」）开头都用了**顶层** `<%_ let _hn; ... _%>` 来读 `路由隐藏规则` 做门控判断，变量名全叫 `_hn`。

**为什么"前100层正常、突然崩"**：这类冲突是"组合触发"型——只有当**两条含 `let _hn` 的条目在同一轮同时被激活**（例如常驻的「行动选项」+ 某条 NSFW 同时命中，或路由脚本把同一片段重复注入）时，一次编译里才会出现两个 `let _hn`。之前一直没凑齐这个组合，所以看起来"一直正常"，本质是**潜伏 bug**。

## 三、正确 vs 错误写法

`_hn` 只是**临时的门控标志**（读变量→`if` 判断→决定这段要不要输出），正文是标签之间的**字面文本**，不存在 `_hn` 里。所以问题纯粹是"变量名在共享作用域里撞了"。

### ❌ 错误：顶层 `let`/`const`（会撞名）

```ejs
<%_ let _hn; try { _hn = JSON.parse(getvar('路由隐藏规则') || '[]'); } catch (e) { _hn = []; } if (!(Array.isArray(_hn) && _hn.includes('[NSFW]'))) { _%>
...正文...
<%_ } _%>
```

### ✅ 首选：用 `{ }` 块级作用域包裹（彻底隔离，可保留 `let`）

```ejs
<%_ { let _hn; try { _hn = JSON.parse(getvar('路由隐藏规则') || '[]'); } catch (e) { _hn = []; } if (!(Array.isArray(_hn) && _hn.includes('[NSFW]'))) { _%>
...正文...
<%_ } } _%>
```

- 开头把 `let` 之前加一个 `{`，结尾对应多补一个 `}`（`if` 的 `}` + 外层块的 `}` = `} }`）。
- 每个 `{ ... }` 是独立作用域，同名变量互不冲突。这是**命定之诗项目一贯采用的写法**，其源码里甚至有注释：*"用大括号括起来的话就不需要担心变量名冲突的问题"*。

### ✅ 次选：改用 `var`（最小改动）

```ejs
<%_ var _hn; ... _%>
```

`var` 是函数作用域、允许同作用域内重复声明，且每块都会重新赋值，逻辑不变。够用，但不如 `{ }` 规范（`var` 仍共享同一个变量槽，只是这里不影响结果）。

> 本次最终统一采用**首选方案 `{ }`**，与命定之诗保持一致。

## 四、`if / else` 结构的包裹方式

「📑行动选项」是 `if/else`，外层大括号要包住**整个 if/else**，`else` 行不动，只改首尾：

```ejs
<%_ { let _hn; ...; if (gate) { _%>
...A...
<%_ } else { _%>
...B...
<%_ } } _%>          ← 收尾：else 的 } + 外层块的 }
```

## 五、排查方法（可复用）

搜"EJS 开标签后**直接**跟 `let/const`、且没有 `{` 包裹"的危险写法（`for (const ...)` 是循环作用域、自动安全，不会被这条命中）：

```bash
grep -rnoE "<%[_-]?\s*(let|const)\s+[A-Za-z_\$][A-Za-z0-9_\$]*" . \
  --include=*.txt --include=*.yaml --include=*.json --include=*.ejs \
  | grep -v node_modules
```

命中后要判断该变量名是否**可能在多条共存条目里重复出现**；即使当前只出现一次（如命定之诗的 `const skillList`），也建议顺手用 `{ }` 包裹，防患于未然。

## 六、检查清单（写新条目时）

- [ ] EJS 里凡是 `let`/`const` 声明，一律用 `<%_ { ... _%> ... <%_ } _%>` 块级作用域包裹。
- [ ] 首尾大括号数量对齐（尤其 `if/else`、`try/catch` 嵌套时，结尾常是 `} }`）。
- [ ] 变量名尽量语义化、避免多条条目撞名（如 `_hn`、`skillList` 这类通用名更要包裹）。
- [ ] 记住：**多条条目共享一个编译作用域**，别假设每条条目是独立作用域。
